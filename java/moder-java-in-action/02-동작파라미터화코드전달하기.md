# 2. 동작 파라미터화 코드 전달하기

### 배경
#### 1. 변화하는 요구사항에 대응하기
    
만약 녹색 사과를 필터링하는 요구사항이 있다면 보통 아래와 같이 구현할 것이다.
```java
enum Color { RED, GREEN }
        
public static List<Apple> filterGreenApples(List<Apple> inventory) {
  List<Apple> result = new ArrayList<>();
  for (Apple apple: inventory) {
    if (GREEN.equals(apple.getColor())) {
      result.add(apple);
    }
  }
  return result;
}
```
녹색 사과 말고 빨간 사과도 필터링하고 싶어진다면? 연두색, 진한 빨간색 등 더 다양한 색의 요구사항이 주어진다면? 그때마다 filterRedApples.. 등을 만들어야 하나?
**요구사항이 늘어날수록 변화에 대응하는 게 어려워진다.**


Try 1. 색을 파라미터화
         
```java
public static List<Apple> filterGreenApples(List<Apple> inventory, Color color) {
  List<Apple> result = new ArrayList<>();
  for (Apple apple: inventory) {
    if (GREEN.equals(apple.getColor())) {
      result.add(apple);
    }
  }
  return result;
}

List<Apple> greenApples = filterApplesByColor(inventory, GREEN);
List<Apple> redApples = filterApplesByColor(inventory, RED);
```
이렇게 한다면 코드가 중복되는 여러 메소드를 생성할 필요가 없다. 하지만 또 색깔 이외에 무게로 필터링 요구사항이 생기면? weight 파라미터를 추가한다고 했을 때  그 다음엔 원산지, 그 다음엔 크기, 그 다음엔 품질 계속해서 파라미터를 추가할 수는 없는 노릇이다.

이렇게 요구사항이 바뀌었을 때 유연하게 대응하기 위해서 **동작 파라미터화**를 이용할 수 있다.

### 동작 파라미터화란?

아직은 어떻게 실행할 것인이 결정하지 않은 **코드 블록** → 이 코드 블록은 프로그램이 나중에 호출함 (코드 블록의 실행은 나중으로 미뤄짐)

#### 예제1.
먼저 선택 조건을 결정하는 인터페이스를 정의한다.
```java
public interface ApplePredicate {
  boolean filter(Apple apple);
}
```

다양한 선택 조건을 대표하는 여러 버전의 ApplePredicate를 정의할 수 있다.

(1) 무거운 사과 필터
```java
public class AppleHeavyWeightPredicate implements ApplePredicate {
  public boolean filter(Apple apple) {
    return apple.getWeight() > 150;
  }
}
```

(2) 녹색 사과 필터
```java
public class AppleGreenColorPredicate implements ApplePredicate {
  public boolean filter(Apple apple) {
    return GREEN.equals(apple.getColor());
  }
}
```
이는 실행 중에 알고리즘을 선택할 수 있게 하는 디자인 패턴 중 `전략 패턴`이라고 할 수 있다.

```java
public static List<Apple> filterGreenApples(List<Apple> inventory, ApplePredicate p) {
  List<Apple> result = new ArrayList<>();
  for (Apple apple: inventory) {
    if (p.filter(apple)) result.add(apple);
  }
  return result;
}
```
메서드가 동작을 파라미터로 받아서 다양한 동작을 수행할 수 있다. 어떤 전략을 사용할지 선택하여 ApplePredicate형의 원하는 필터를 파라미터로 넣어주면 되는 것이다.


이는 컬렉션(예제에서는 inventory)을 탐색하는 로직과 각 항목에 적용할 동작(예제에서는 필터링)을 **분리** 할 수 있다는 것이 바로 동작 파라미터화의 가장 큰 장점이다.

여기서 문제점: 동작을 추가할 때마다 클래스를 추가해줘야 하는 번거로움 발생

### 개선
#### 1. 익명 클래스
익명 클래스란 말 그대로 이름이 없는 클래스로 이를 활용하면 클래스 선언과 인스턴스화를 동시에 할 수 있다.
```java
List<Apple> redApples = filterApples(inventory, new ApplePredicate() {
  public boolean filter(Apple apple) {
    return RED.equals(apple.getColor());
  }
});
```
메서드의 동작을 직접 파라미터화 한 것을 볼 수 있다. 하지만 클래스 선언 과정만 줄여줄 뿐 **여전히 많은 공간을 차지 한다는 문제점**이 있다. 
#### 2. 람다식
Java8부터 등장한 개념으로
```java
List<Apple> redApples = filter(inventory, (Apple apple) -> RED.equals(apple.getColor()));
```
익명 클래스보다 훨씬 간단하고 가독성도 좋다.
#### 3. 리스트 형식으로 추상화
```java
public interface Predicate<T> {
  boolean test(T t);
}

public static <T> List<T> filter(List<T> list, Predicate<T> p) {
  List<T> result = new ArrayList<>();
  for(T e: list) {
    if(p.test(e)) result.add(e);
  }
  return result;
}
```

이렇게 제네릭타입을 활용해 리스트 형식으로 추상화하면 훨씬 더 유연하게 사용할 수 있다.
```java
List<Apple> redApples = filter(inventory, (Apple apple) -> RED.equals(apple.getColor()));
List<Integer> evenNumbers = filter(numbers, (integer i) -> i % 2 == 0);
```
사과만 필터링할 수 있는 게 아니라 정수, 문자열 등 다양하게 필터링할 수 있게 확장 가능하다.
### 활용
이런 동작 파라미터화는 실전에서 `Comparator` 정렬 `Runnable` `Collable` `EventHandler` 에서 활용되고 있음을 확인할 수 있다.
#### 1. Comparator 정렬
`java.util.Comparator` 객체를 이용해서 정렬을 파라미터화 할 수 있다. `Comparator`은 인터페이스로 `compare` 메소드를 구현해줘야 하는데

```java
inventory.sort(new Comparator<Apple>() {
  public int compare(Apple a1, Apple a2) {
    return a.getWeight().compareTo(a2.getWeight());
  }
});
```
요구사항이 바뀌면 새로운 요구사항에 맞는 `Compartor`를 만들어 sort 메서드에 전달하면 된다.

단일 추상 메서드로 람다 표현식을 활용해서 더 간략하게 표현할 수 있다.
```java
inventory.sort((Apple a1, Apple a2) -> a1.compareTo(a2.getWeight()));
```
이게 가능한 건 `List` 인터페이스의 `sort` 메서드가 다음과 같이 구현되어 있기 때문이다. 즉 sort()는 `Comparator` 타입을 인자로 받는 메서드다.
```java
void sort(Comparator<? super E> c);
```
#### 2. Runnable
`Runnable`을 이용하면 다양한 동작을 스레드로 실행할 수 있다.

```java
Thread t = new Thread(new Runnable() {
  @Override
  public void run() {
    System.out.println("hello world");
  }
});
```

`Runnable` 역시 추상 메서드는 `run()` 하나뿐이기 때문에 역시 람다식으로 간단하게 표현 가능하다. (단일 추상 메서드)
```java
Thread t = new Thread(() -> {
  System.out.println("hello world");
});
```
#### 3. Collable
`Collable` 역시 병렬 실행을 위해 사용하는 것인데 `Runnable`과 비슷하게 구현된다. 
#### 4. EventHandler
자바 FX에서는 `setOnAction` 메서드에 `EventHandler`를 전달함으로써 이벤트에 어떻게 반응할지 설정할 수 있다.
```java
Button button = new Button("Send");
button.setOnAction(new EventHandler<ActionEvent>() {
  public void handle(ActionEvent event) label.setText("Sent!!");
});
```
역시 람다 표현식으로 구현할 수 있다.

```java
button.setOnAction((ActionEvent event) -> label.setText("Sent!!"));
```

#### 결론
Java는 원래도 객체 지향 언어이지만 Java8이 등장하면서 람다가 생기고 위에서 알아본 것처럼 동작 전달 비용이 크게 줄어 들었다. 즉 `람다`와 `Functional Interface`가 **동작 파라미터화를 구현하는 핵심 도구**임을 알 수 있다.
